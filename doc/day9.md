# 第9日：テスト自動化の基礎

本日は、研修の締めくくりとして、現代のプロジェクトで欠かせない**「テスト自動化」**について学びます。テスターとして「自動化とどう向き合い、どう活用するか」という実務的な視点を整理しましょう。

---

## 1. テスト自動化とは

テスト自動化とは、これまで手動で行っていたテスト実行を、コンピューター（スクリプト）に代行させる仕組みのことです。テスターにとっては、 **「繰り返しの作業を機械に任せ、人間はより高度な確認に集中するための手段」** と言えます。

## 2. 何を自動化すべきか？（テスト戦略）

すべてのテストを自動化するのは、コストや手間の面で現実的ではありません。手動テストと自動化の使い分け（戦略）を理解することが重要です。

| 項目 | **手動テスト (Manual)** | **自動テスト (Automation)** |
| :--- | :--- | :--- |
| **得意なこと** | UIの微調整、直感的な違和感の検知、使いやすさ（使用性）の評価。 | 大量のデータ処理、全く同じ操作の繰り返し、高速な検証。 |
| **向いている対象** | **頻繁に仕様が変わる画面**、新機能の初回確認。 | **安定した機能・API**、データベースの整合性チェック、**回帰テスト**。 |
| **テスターの役割** | ユーザーの視点で「攻めのテスト」を行う。 | 正確な「テスト手順（設計）」を自動化の基盤として提供する。 |

## 3. 自動化の最大の目的「回帰テスト（リグレッションテスト）」

システムに修正や機能追加をした際、**「今まで動いていた別の場所」が壊れていないか**を確認するテストを**回帰テスト**と呼びます。
    現代の複雑なシステムでは、一箇所を直すと関係ない場所が壊れる「デグレード（先祖返り）」が起きやすいため、このテストが非常に重要です。
    回帰テストを自動化することで、開発のたびに全機能の健全性をボタン一つで確認できるようになり、人的なミスや工数を劇的に削減できます。

## 4. 自動化におけるテスターの貢献

自動化スクリプトを書くのはエンジニアの役割（テストエンジニア等）となることが多いですが、**「何を・どのような手順で・どんな期待値でテストするか」という設計図を作るのはテスターの仕事**です。

**正確な手順の記述：** 誰が（あるいは機械が）実行しても同じ結果が出る客観的な手順書を作成します。

**データの準備：** 自動化に適した「テスト用マスターデータ」を定義します。

**結果の判定：** 自動化が「NG」を出した際、それが「システムのバグ」なのか「自動化スクリプト自体の不備」なのかを判断し、適切に報告します。

## 5. 自動化によって生まれる「余裕」の活用

単純な繰り返しテストを自動化に任せることで、テスターは以下のような**人間にしかできない価値の高い活動**に時間を割けるようになります。

**アドホックテスト（探索的テスト）：** 仕様書にない「もしも」の操作や、実行中の「違和感」を追求するテスト。

**ユーザー視点の検証：** 「仕様通りだが、本当に使いやすいか？」という満足度の追求。

**リスクの高い箇所の深掘り：** バグが潜みやすい境界値や複雑なロジックへの集中。

---

## まとめ

テスト自動化は、テスターの仕事を奪うものではなく、**「より質の高いテスト」を実現するための強力なパートナー**です。
8日間の研修で学んだ「正確な観点出し」や「論理的なテスト設計」は、自動化を成功させるための最も重要な土台となります。機械に任せられる部分は任せ、皆さんは「ユーザーの代弁者」として、人間にしか見抜けないリスクを見つけ出すプロフェッショナルを目指してください。
