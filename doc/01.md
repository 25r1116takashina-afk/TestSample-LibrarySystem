# 第1日：テストの基礎とQAの本質

本日は、テスターとしての第一歩です。技術的な手法を学ぶ前に、私たちが「何のためにテストをするのか」という**マインドセット（考え方）**をしっかりと身につけましょう。

## 1. ソフトウェアテストの必要性

現代社会において、ソフトウェアはインフラの一部です。もしシステムが止まったり、誤作動したりすれば、その影響は単なる「不便」では済まされません。

| 影響範囲 | 具体的なリスク例 |
| --- | --- |
| **経済的損失** | 銀行システムの停止、ECサイトでの二重決済、誤った価格表示。 |
| **社会的信用の失墜** | 個人情報の流出、サービス長期停止によるブランドイメージの低下。 |
| **生命・安全への脅威** | 医療機器の誤作動、自動運転の判断ミス、航空管制システムのトラブル。 |

**「バグは必ず存在する」**という前提に立ち、最悪の事態を未然に防ぐのがテストの役割です。

---

## 2. 品質保証（QA）とテストの違い

よく混同されますが、「テスト」と「QA（Quality Assurance）」は範囲と目的が異なります。

| 項目 | テスト（Test / QC） | 品質保証（QA） |
| --- | --- | --- |
| **定義** | 成果物を動かし、不具合を探す活動。 | 望ましい品質を定義し、それを保証する仕組み作り。 |
| **視点** | 「この機能は正しく動くか？」 | 「このプロセスで、ユーザーは満足するか？」 |
| **アプローチ** | **事後的・対症療法的**（見つかったバグを直す）。 | **予防的・プロセス指向**（バグが出ない工程を作る）。 |
| **役割** | チェック担当（テスター）。 | 全体の守護神（QAエンジニア）。 |

> **Point:** テスターはQAという大きな枠組みの中の、非常に重要な「実行部隊」です。

---

## 3. テストの目的

テストの目的は、単にバグをゼロにすることではありません（現実的に、バグをゼロにすることは不可能です）。真の目的は以下の2点に集約されます。

### ① リスクの低減

システムをリリースした後に発生する「致命的な問題」の可能性を、テストによって事前に下げます。

* **不具合の早期発見：** 開発の初期段階で見つけるほど、修正は容易で安価になります。
* **未実施事項の可視化：** 「どこをテストしていないか」を明確にすることも、リスク管理の一つです。

### ② 信頼性の向上

「このシステムは、これだけのテストをクリアしたから安心だ」という**根拠（エビデンス）**を提供します。

* **仕様への適合：** 要求された通りに作られているかを確認します。
* **ユーザー体験の保証：** 操作性や分かりやすさなど、数値化しにくい満足度も対象となります。

---

## 4. テストエンジニアの心得

テスターが陥りやすい「間違ったマインド」と「正しいマインド」を確認しましょう。

* **✕ 間違ったマインド：**
* 「バグを見つけるのが仕事だから、開発者のミスを指摘して優位に立とう」
* 「仕様書通りに動けば、ユーザーが困っても関係ない」


* **◯ 正しいマインド：**
* 「開発チームのパートナーとして、一緒に最高の製品を作り上げよう」
* 「ユーザーがどう使うかを想像し、仕様書の行間にあるリスクを読み取ろう」
